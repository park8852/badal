## 2. 개요 (슬라이드 3~5)
### 2-1. 프로젝트 개요
> “저희 팀은 모의해킹 대상을 직접 구축하면서 개발 역량을 강화하고, 구축한 시스템을 분석해 취약점 분석 능력을 키운 뒤, 그 결과를 바탕으로 공격 시나리오를 설계 및 실행함으로써 **모의해킹 수행 역량까지 종합적으로 향상**시키는 것을 목표로 하고 있습니다.”

### 2-2. WBS
> “WBS는 기획, 개발, 취약점 진단, 모의해킹, 자동화 다섯 단계로 나누어 **1~3주 단위**로 계획했습니다. 현재는 취약점 진단까지 완료했고, 앞으로 모의해킹과 자동화 스크립트 개발을 진행할 예정입니다.”

### 2-3. 팀 소개
> “팀명은 **남탕**이고 총 7명입니다.  
> - 변창민 교육생: PM, 백엔드, 웹 진단  
> - 이유준 · 임지혁 · 박찬희 교육생: 프론트엔드, 웹 진단  
> - 박진영 교육생: 프론트엔드, 모바일 진단  
> - 장상훈 교육생: 백엔드, 모바일 진단, 시나리오 1  
> - 손성민 교육생(발표자): 모바일 개발, 모바일 진단, 시나리오 2

## 3. 서비스 소개 (슬라이드 6~8)
### 3-1. 서비스 선정 이유
> “최근 **배달 플랫폼들의 개인정보 유출 이슈**가 빈번했습니다. 배달의민족, 요기요, Yes24 등에서 고객 정보가 유출된 사례가 있었고, 이런 환경에서 배달 서비스는 **개인정보와 결제 정보가 교차**하는 중요한 대상이라고 판단했습니다.”

### 3-2. 취급 데이터와 위험 교집합
> “일반적으로 개인정보에는 이름·생년월일, 휴대전화번호, 상세 주소, 결제/카드 정보, 위치 정보 등이 포함됩니다. 우리 서비스 ‘바로배달’에서는 회원가입, 주문 정보, 결제, 알림 기능을 통해 이 데이터들이 모두 수집·사용됩니다.  
> 결합 시 다음과 같은 치명적 위험이 발생할 수 있습니다.  
> - 전화번호 + 주소 → **신원 도용·스토킹**  
> - 인증 토큰 → **계정 탈취**  
> - 결제/카드 정보 → **무단 결제**  
> - 위치·주문 이력 → **프라이버시 침해·피싱**  
> 이러한 교집합 위험을 다루기 위해 배달 서비스를 모의해킹 대상으로 선정했습니다.”

(타이머 참고: 약 2분 20초)

## 4. 개발 진행 현황 (슬라이드 9~24)
### 4-1. 기술 스택
> “사용자 앱은 **Android Jetpack Compose (Kotlin)**, 웹 프론트는 **Next.js (TypeScript)**, 백엔드는 **Spring Boot (Java)**, DB는 **MariaDB**, 인프라는 AWS로 구성했습니다.”

### 4-2. 역할 기반 기능
> “클라이언트를 **사용자 / 가게 사장(사업자) / 관리자**로 구분했습니다.  
> - 사용자: 회원가입·로그인 후 가게 탐색, 메뉴 선택, 주문, 포인트 결제  
> - 사업자: 가게 등록, 메뉴 관리, 주문 조회, 월 매출 확인  
> - 관리자: 사용자·사업자 관리, 주문·가게 삭제, 매출 정산, 공지·Q&A 관리”

### 4-3. 인프라 & 아키텍처
> “클라우드 구성도에서 웹 프런트 서버는 DMZ, WAS와 DB는 프라이빗 망에 배치했습니다.  
> 아키텍처는 **클라이언트 → 백엔드 → 인증 계층 → 데이터 계층**으로 설계했고, 주문·사용자·가게·매출 데이터를 MariaDB에 저장합니다.”

### 4-4. API & DB
> “REST API는 주문, 메뉴, 가게, 포인트, 인증 5가지 카테고리로 총 28개 엔드포인트를 구현했습니다.  
> DB는 사용자, 가게, 메뉴, 주문, 매출 5개 테이블로 구성되어 있으며 외래 키로 유기적으로 연결됩니다.”

### 4-5. UI 시연 (모바일/웹/관리자)
> “모바일 앱에서는 로그인·회원가입·메인, 마이페이지(공지·문의·포인트 조회), 포인트 충전, 메뉴 주문, 주문 내역 등을 구현했습니다.  
> 사업자 웹은 회원가입/로그인 후 마이페이지, 주문 관리, 가게 관리(카테고리·운영시간·로고), 메뉴 CRUD, 매출 관리(기간별 매출), 공지사항 열람 기능을 제공합니다.  
> 관리자 웹은 전체 고객·가게·주문·매출 요약과 함께 가게 삭제, 주문 삭제, 가게별 매출 정산, 공지/Q&A 관리를 지원합니다.”

## 5. 취약점 진단 (슬라이드 25~31)
### 5-1. 진단 개요
> “금융보안원 ‘금융정보취급시스템 취약점 분석평가(금취분평)’ 기준을 토대로 웹과 모바일 항목을 분리해 진단했습니다.”

### 5-2. 웹 진단 결과
> “웹에서는 21개 체크리스트 중 **9개 취약, 12개 양호**를 확인했습니다.  
> 취약 항목은 고정된 인증정보, 불충분한 이용자 인증, 무제한 요청 허용, 브라우저 중요정보 노출, CSRF, 인증 오류 횟수 제한 부재, 세션 재사용, 인증수단 소유자 검증 미흡, 통신구간 암호화 미적용 등입니다.”

### 5-3. 모바일 진단 결과
> “모바일 쪽은 35개 항목 중 **20개 취약, 15개 양호**를 확인했습니다. 특히 전자금융 항목에서 거래정보 재사용, 거래 시 소유주 확인, OS 변조 탐지, 악성코드 방지, 입력정보 보호, 프로그램 무결성 검증, 소스코드 난독화 등이 취약으로 나타났습니다. 양호 항목에는 디버깅 탐지 기능 적용 등이 있습니다.”

## 6. 시나리오 기반 모의해킹 (슬라이드 32~34)
### 6-1. 시나리오 1 – 협력사 사칭 피싱을 통한 핵심 서버 장악
> “시나리오 1은 **협력사 사칭과 공급망 공격**을 통해 내부 개발자 PC를 장악하고, 이를 통해 백엔드 서버와 DB까지 침투하는 시나리오입니다.  
> 1) APK 분석 → AppSuite 사용 파악  
> 2) LinkedIn에서 개발자 A의 명함 정보 수집  
> 3) 스틸리언 직원으로 위장해 패치 프로그램 전달 → 실행 시 리버스 쉘 확보  
> 4) 키로거 설치, 프로세스·환경 변수 수집, 크롬 프로필 삭제로 재로그인 유도  
> 5) 탈취한 구글 계정으로 Gmail 접근 → 백엔드 자격증명(AWS 키 등) 확보  
> 6) EC2 접속, 환경변수에서 DB URL/ID/PW 탈취 → DB 덤프 유출  
> 피해는 개인정보 대량 유출과 서비스 마비 가능성이 큽니다.”

### 6-2. 시나리오 2 – AI 인젝션과 LangFlow 취약점을 통한 내부망 장악
> “시나리오 2는 **AI 프롬프트 인젝션 + LangFlow 취약점(CVE-2025-3248 가정)**을 활용해 내부망을 장악하는 시나리오입니다.  
> 1) VPN을 통해 사내 인트라넷 웹에 접근 → 보안 미적용 챗봇 확인  
> 2) 프롬프트 인젝션으로 내부 직원/시스템 정보를 탈취  
> 3) 챗봇이 연동된 AI Backend(WAS) 주소, 버전을 확인하고 LangFlow 1.2.0 취약점 공격  
> 4) WAS 장악 후 내부망 스캔 → 내부 DB·Vector DB 접근 → 자료 유출 및 랜섬웨어 전개  
> 피해는 내부 자료 탈취, AI 모델 오염, 내부망 마비로 이어질 수 있습니다.”
